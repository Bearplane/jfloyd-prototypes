Q-ScanAI Scanner Configuration

Three-tier config (Industry Type / Type of Company / Company) · Overview + Edit UX · Additional Words (Compliance Elements) · Many-to-one data model requirement.

⚠ Data Model Requirement — Many-to-One (Post-MVP Priority)
A single company can belong to multiple Industry Types and/or Types of Company simultaneously.
Currently the system is one-to-one — one Industry Type per company. For this feature to work correctly in real-world scenarios, we need a many-to-one relationship: one company maps to multiple Industry Types and/or multiple Types of Company. The scanner config for that company must aggregate all unique settings from every matching tier config, then apply any company-level overrides on top.

This does not need to ship in the initial release but will be critical for handling mixed-category merchants (e.g., a retailer selling both CBD products and firearms). The initial v1 can use a single Industry Type assignment per company; the many-to-one aggregation is a fast-follow requirement.

Aggregated Scanner Config — what the scanner actually uses
Checks: Union of all unique checks from Hemp/CBD baseline + Firearms & Ammo baseline + E-commerce override (Accepted Payment Methods added)
Custom Keywords: Union of all unique keywords from Hemp/CBD config + Firearms & Ammo config
Whitelist Keywords: Union of all unique whitelist terms from all matching tiers
Additional Words: Union of all violative term categories — Peptide Compliance, Drug References, Health Claims, Cannabis Compliance (from Hemp/CBD tier) + Firearms Compliance (from Firearms & Ammo tier)
Conflict resolution for boolean checks: If a check is enabled in any matching tier config, it is enabled in the aggregated config. Company-level overrides can explicitly disable a check for a specific merchant. For keywords and Additional Words: additive union — no deduplication needed since the scanner checks for presence of any term.
① Scanner Tab — What the User Sees
Overview (default) and Edit Configuration toggle
Design decision — Edit toggle in the Scanner tab itself, not a separate Settings tab.
The user is already in the Scanner tab when they're about to run a scan. Keeping the loaded-config summary and the "Edit configuration" button in the same view means they can review what config will be used, make any needed tweaks, and immediately start the scan — without navigating away. A separate Settings tab would require context-switching back and forth. Recommended approach: Scanner tab with Edit toggle.

② How it resolves
Lookup order at scan time
Company override → falls back to → Type of Company → falls back to → Industry Type → falls back to → Existing per-company default

If a company has no override saved, the scanner config inherits from the Type of Company tier. If no Type of Company tier is saved, it inherits from the Industry Type tier. If nothing is saved at any tier, the existing per-company default applies — same as today.

With many-to-one (post-MVP): A company mapped to multiple Industry Types and/or Types of Company gets an aggregated config — the union of all unique checks, keywords, and Additional Words from all matching tier configs, with Company-level overrides applied last at highest priority.
