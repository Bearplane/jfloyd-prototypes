Q-Config — Feature Limits Workflow

End-to-end walkthrough of the Variables / Limiting Factors / Triggers / Plans model that drives MyCOA's Product Limit (and every other plan-tiered constraint going forward). Improves the existing Feature Builder placeholder UI so the model becomes legible and the four concepts read as deliberate, sibling building blocks.

Q-Config · Feature Builder + Plan Builder Improves existing placeholder UI First consumer: MyCOA productLimit Initially paywall filtering only — no UI blocking
How to read this mockup Yellow dashed blocks are review notes — never rendered in-app. The Feature Builder modal already exists in placeholder form (Details / Sources / Limiting Factors / Variables tabs, capacity values rendered as "5, 10, 25" comma-strings). This mockup keeps that 4-tab shape but improves the editing affordance: capacity tiers become chips/pills, an explicit "Unlimited" chip captures the blank-means-unlimited semantic, the typo "Limiting Fators" is fixed, and the primary action button is corrected from green to BLACK #1A202E per the design system. Each tab gains a one-line help banner that explains what the tab captures and how it relates to the others.
  • Variables are just field defs on a Feature: title, description, code, type, default. They never enforce anything on their own — they're the lexicon a Feature exposes for its Limiting Factors and Triggers to consume.
  • Limiting Factors are enforcement. A Limiting Factor wraps one Variable with a list of capacity tiers. Plans pick a tier; the system enforces (or — for MyCOA productLimit — filters paywall plans by) that tier.
  • Triggers are soft nudges. Same architecture but the Variable powers a UI message ("you've used 80% of your weekly scans") instead of a hard limit.
  • Plans reference Features and pick a capacity tier from each of that Feature's Limiting Factors. A Feature added to a plan with no tier picked = unlimited by default.

Architecture summary — four concepts, one workflow

Read this first
⌬

Variable

The "what"

A field a Feature exposes. Just a definition — title, description, code, data type, default value. Doesn't enforce anything by itself.

MyCOA examples: productLimit, currentProductCount, maxScanCount, currentScanCount
⛔

Limiting Factor

The "enforcement"

Wraps a Variable with capacity tiers. Plans pick one tier; the system uses it to gate, filter, or count.

Product Limit: productLimit + tiers [100, 250, 500, Unlimited]
⚡

Trigger

The "soft nudge"

Watches a Variable and fires a UI message at a threshold. No blocking — just upgrade prompts and warnings.

Scans nearing cap: watch currentScanCount ≥ 80% of maxScanCount → "consider upgrading"
▣

Plan

The "subscription"

References Features and picks one capacity tier from each Limiting Factor those Features expose. Drives paywall + monitoring.

Limiting Factor Definition — Feature Builder · Limiting Factors tab

Q-Config admin · MyCOA Feature
STEP 02
Same Feature, the Limiting Factors tab. The admin creates a Limiting Factor by picking a Variable and adding capacity tiers as chips. Each chip is one selectable value a plan can choose. The special Unlimited chip is the explicit form of "blank/NULL means unlimited" — making the semantic visible instead of hidden in absence-of-value. The Product Limit Limiting Factor is the worked example: Variable = productLimit, tiers = [100, 250, 500, Unlimited].

Trigger Definition — Feature Builder · Triggers tab

Q-Config admin · MyCOA Feature · sibling concept (soft nudges)
STEP 03
Triggers are the soft-nudge cousin of Limiting Factors. Same wiring (a Trigger consumes a Variable) but instead of enforcing a hard cap, it fires a UI message when a threshold is reached. The placeholder Feature Builder doesn't have a Triggers tab yet — Joseph mentioned this as the natural other consumer of the Variables system. Adding it as a fifth tab keeps the architecture symmetric and gives the team a place to wire the "you've used X of Y" copy.

Triggers fire UI nudges, not hard caps. A Trigger watches a Variable and when it crosses a threshold (often expressed relative to a second Variable, e.g. 80% of maxScanCount), it surfaces a message in-app. Useful for upgrade prompts, capacity warnings, and re-engagement. Independent of Limiting Factors — you can have a Trigger without a hard cap, and vice-versa.

Plan Editor — adding MyCOA, picking capacity tiers

Q-Config · Subscription Builder · MyCOA Pro plan
STEP 04
An admin builds a Plan and adds the MyCOA Feature to it. As soon as a Feature is added, every Limiting Factor defined on that Feature surfaces as a tier picker. The admin picks one chip per Limiting Factor; that's how the Plan binds to capacity. For MyCOA Pro: Product Limit = 250, Weekly Scans = 25. No tier picked = unlimited (interpreted from the absence of a saved tier on the plan).

Why this beats the placeholder: the placeholder Subscription Builder lets admins type a number into a flat field per limit — no tie-back to which Variable / Limiting Factor it reflects, and no enforcement that the value matches an authored tier. Merchant paywall — filtering by Product Limit at onboarding

Onboarding wizard · merchant has 200 products
STEP 05
A new merchant onboards with 200 products they want to bring into MyCOA. The paywall reads the merchant's declared inventory size, queries every published plan's MyCOA productLimit tier, and hides plans whose tier is below 200. The merchant sees only the plans they could actually use, ordered cheapest-first. Their inventory size becomes a recommendation signal, not a friction point.

Internal · per-merchant detail · Finances tab
STEP 06

No UI blocking — purely monitoring data so the team can spot merchants outgrowing their tier and reach out manually.

Data model recap — what the dev team builds against

For the eventual QP ticket

Persisted shape (simplified)

Variables / Limiting Factors / Triggers live on a Feature. Plans bind to capacity through feature_limits.

// features table — one row per Feature
{
  slug: "mycoa",
  name: "MyCOA",
  variables: [
    { code: "productLimit",        title: "Product Limit",         type: "number", default: null },
    { code: "currentProductCount", title: "Current Product Count", type: "number", default: 0    },
    { code: "maxScanCount",        title: "Max Scan Count",        type: "number", default: null },
    { code: "currentScanCount",    title: "Current Scan Count",    type: "number", default: 0    }
  ],
  limitingFactors: [
    {
      title: "Product Limit",
      variable: "productLimit",
      capacityTiers: [100, 250, 500, null]   // null === Unlimited
    },
    {
      title: "Weekly Scans",
      variable: "maxScanCount",
      capacityTiers: [5, 10, 25, 100, null]
    }
  ],
  triggers: [
    {
      title: "Scans nearing weekly cap",
      watch: "currentScanCount",
      threshold: ">= 0.8 * maxScanCount",
      message: "You've used {{currentScanCount}} of {{maxScanCount}} weekly scans..."
    }
  ]
}

// subscription_plans table — one row per Plan, references Features by slug
{
  slug: "mycoa-pro",
  name: "MyCOA Pro",
  features: ["mycoa"],
  feature_limits: {
    mycoa: {
      productLimit: 250,        // must match an authored tier on the LF
      maxScanCount: 25
    }
  }
}

// paywall filtering query (pseudocode)
plans
  .filter(p => p.features.includes("mycoa"))
  .filter(p => {
    const tier = p.feature_limits.mycoa.productLimit;
    return tier === null || tier >= merchant.declaredProductCount;
  })
  .sort((a,b) => a.price - b.price);
Out of scope for MyCOA launch · noted for the followup ticket
  • • Server-side enforcement of productLimit writes — initially soft. Per Q-Config truth doc: product/COA limits aren't reliably enforced server-side today, and middleware auto-create paths can bypass UI assumptions.
  • • Trigger delivery surface — where the message lands (in-app banner vs. email vs. push). Mockup shows the copy; the channel is a separate decision.
  • • Tier validation at plan-save time — the plan editor radio chips constrain admins, but a backend check should reject any feature_limits value that isn't an authored tier on the LF.
  • • Migration path from the existing flat subscription_plans.metadata limits onto the new feature_limits shape — one-time backfill.
