Product-Derived Signal Suggestions

For companies that have products, Q-ScanAI infers additional scan-config suggestions from product families and Named Classifications. These are additive only — they extend the resolved config beyond what the Industry-pair template provides, but never override or remove it. An analyst accepts a suggestion to add it as an override; ignored suggestions stay listed for future review. For companies with no products (or Industry pairs without a relevant catalog), this strip simply doesn't appear.

§CInference logic · how a suggestion is generated

For engineers and analyst leads. The suggestion engine reads three inputs and produces ranked tiles. Confidence is bucketed HIGH / MED / LOW based on coverage + source authority — analysts see the bucket inline.

// inputs per company
const products      = company.products
const classifications = products.flatMap(p => p.classifications)   // L3
const pair_templates  = company.industry_pairs.map(loadTemplate)

// catalog → suggestion mapping (table-driven, see signal_inference_rules)
for (rule of signal_inference_rules) {
  const matches = products.filter(rule.match)
  if (matches.length >= rule.min_coverage) {
    if (pair_templates.some(t => t.has_signal_on(rule.target_signal))) {
      // already on via pair template — no suggestion needed
      continue
    }
    suggest({
      signal:     rule.target_signal,
      confidence: confidenceFor(matches.length, rule.expected_coverage),
      evidence:   matches.map(m => m.id),
      additive:   true,                // never produces an OFF
    })
  }
}
Key invariants. Product signals never produce an OFF — only ON suggestions. Pair templates always run first; signals add on top. Acceptance writes a per-company override (layer 4) tagged with source: product_signal so the audit log distinguishes manual overrides from accepted-suggestion overrides. Re-running inference is idempotent — already-accepted suggestions are skipped, dismissed ones reappear (with a reason note) only if evidence changes.
