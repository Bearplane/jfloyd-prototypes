MyCOA Shared Workflow — where we stand today, what’s being asked, and the best path forward

This board is meant to help the full team make a logical decision. It separates the current platform reality from the requested future-state, shows the real blockers, compares the available options, and recommends the path that balances speed, honesty, and architecture risk.

Current truth
MyCOA is still effectively single-company scoped
Requested future state
Wizard-first onboarding plus a shared merchant / processor workflow
Real blocker
Data ownership, permissions, and entitlement enforcement — not just wizard UI

What leadership should know

We can ship a meaningful wizard faster than we can safely redesign shared data ownership.
The requested collaboration model assumes capabilities the platform does not cleanly have today.
Paywall / plan limits are not robustly enforced enough yet to treat them as solved.

Decision the team actually needs to make

Are we building a wizard for merchant-owned MyCOA setup?
Or are we building a true merchant / processor shared workspace?

1. Where the platform stands today

The current system already supports pieces of the vision, but not the full collaborative behavior being requested.

Current-state architecture

What exists today

  • Products and COAs are owned by one company via pl_company_id.
  • Merchant Processors can monitor and annotate client-owned data in parts of Q-Monitor.
  • MyCOA has publishing / display behavior and viewer configuration.
  • Plan catalog, paywall flags, pricing, and some gating behavior already exist.
  • Invite, subscription, company relationship, and scanner systems all exist — but as separate flows.

What does not exist cleanly today

  • No true shared merchant / processor workspace around MyCOA.
  • No fine-grained permission model like “can edit but not delete” or “can publish but not archive.”
  • No robust server-side enforcement for product / COA limits in line with plan pricing.
  • No clean invite → wizard → tailored paywall → collaborative workflow pipeline.
  • No MP-aware MyCOA operating model.

Today’s ownership model

Merchant-owned products and COAs. Processor visibility comes through monitoring / overlay behavior, not co-ownership.

→

Requested experience

Invite the merchant, run a setup wizard first, then show a tailored paywall, then let both sides operate in a coherent workflow.

→

Why that breaks

The requested flow assumes shared permissions, shared operating context, and plan-limit enforcement the platform does not cleanly provide today.

2. The core blocker

These are the four architecture questions hiding under the wizard discussion.

Blocker map

Data ownership

Does the merchant remain the canonical owner of products and COAs, or do we introduce a new shared workflow layer?

Permissions

The current role model is too coarse for the requested future-state. We do not yet have relationship-aware action permissions.

Invite / onboarding

The current invite flow does not naturally become a wizard-first, plan-aware, processor-contextual onboarding experience.

Entitlement enforcement

Product and COA limits are not robustly server-side enforced today, so tailored plan selection has loophole risk.

MyCOA scope

MyCOA is still company-scoped, not processor-aware. That matters if the requested future-state assumes collaboration in one shared place.

QP-3226 collision risk

If we harden the wrong isolation model too early, we may make the requested future-state much harder to build cleanly later.

3. Options moving forward

These are the three viable paths based on the architecture we actually have today.

Option comparison
Option A

Wizard-first onboarding only

Build the quick-start wizard and tailored paywall inside the current ownership model. Merchant remains owner. Processor visibility stays in existing monitoring patterns.
Fastest Lowest risk
+Most honest to the current platform
+Gets a real deliverable in front of merchants faster
~Does not solve true processor / merchant co-working
Option B

Build the full shared workflow now

Redesign the platform to support a true merchant / processor shared workflow now, including permissions, onboarding, and entitlement logic.
Highest risk Largest scope
+Closest to the fully requested future-state
−Touches ownership, permissions, invites, and plan enforcement all at once
−Easiest path to accidental architecture debt if rushed
Option C · Recommended

Phase it: wizard now, shared model after alignment

Ship a meaningful wizard-first onboarding flow in the current model, then deliberately decide and design the shared merchant / processor workflow as a second architectural track.
Balanced speed Lower architecture risk
+Delivers progress now without pretending the platform already supports the full vision
+Creates a clean decision boundary for QP-3226 and permissions work
~Requires discipline to keep Phase 1 honest and not sneak in half-shared behavior

Recommendation

Option C is the best path. Ship the MyCOA setup wizard as a wizard-first onboarding flow inside the current ownership model, and treat the full shared merchant / processor workflow as a separate architecture decision that must be resolved before it is promised as part of the same deliverable.

This gives the team something real to build now while staying honest about the fact that data ownership, permissions, and entitlement enforcement are not settled enough yet for the full requested future-state.

What to say in the meeting

  • We can move fast on the wizard itself.
  • The shared workflow is the part that needs architecture alignment.
  • The current system supports monitoring overlays, not true shared co-ownership.
  • Plan / paywall logic is not strong enough yet to treat limits as solved.
  • So the honest move is to split delivery from architecture decision-making.

4. What would need to be resolved before the full requested option is safe

If the team wants the full shared workflow later, these are the prerequisites.

Pre-work required
1

Ownership model

Decide whether merchant remains canonical owner or whether a true shared workspace model is required.

2

Permission model

Define relationship-aware actions like edit, delete, publish, review, and approve.

3

Invite flow

Define what the invite creates, when wizard happens, when paywall happens, and where users land afterward.

4

Limit enforcement

Decide what count matters, when it is checked, and how product/COA loopholes are prevented.

5

QP-3226 reframing

Make sure data-isolation work does not accidentally harden a model that blocks the desired future-state.

5. Friendly talking points for the team

Use these to frame the discussion without forcing a premature technical decision.
