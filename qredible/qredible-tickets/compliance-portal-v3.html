The Vertical Slice — what gets built end-to-end

Findings from any of three sources can become a case. Cases land in Internal Case Management for review and send-out. Merchants log in, respond, and resolution flows back to the processor. Closing the loop is the v1 success criterion.

Engineer · Loop closure state transitions
// Processor "Accept Response":
POST /api/qmonitor/cases/:id/responses/:itemId/accept
  Side effects (atomic):
    1. response.status: SUBMITTED → ACCEPTED
    2. item.status: SUBMITTED → RESOLVED
    3. if all items.RESOLVED → case.status: IN_REVIEW → RESOLVED
    4. emit ActivityEvent { type: "RESPONSE_ACCEPTED", caseId, itemId, ts }
    5. emit ActivityEvent { type: "CASE_RESOLVED", caseId, ts }   // if case flipped
    6. notify merchant (in-portal banner; email optional v2)

// Processor "Request More Information":
POST /api/qmonitor/cases/:id/responses/:itemId/reject
  Body: { reason: string }
  Side effects:
    1. response.status: SUBMITTED → REJECTED
    2. item.status: SUBMITTED → PENDING (back to merchant)
    3. emit ActivityEvent { type: "RESPONSE_REJECTED", reason }
    4. notify merchant in-portal

// v1 deliberately does NOT close case automatically on resolve.
// Force Close (QP-3591) defers manual close to v2; v1 has explicit "Close Case" button on Items tab when all RESOLVED.

G.1 · Core entities (v1 shape)

PlComplianceCase
{
  id: string // "CR-2026-0042"
  title: string
  merchantId: string
  processorOrgId: string // "flex-payment-solutions"
  priority: "HIGH" | "MEDIUM" | "LOW"
  status: CaseStatus // see G.2
  slaDeadline: ISO-8601
  assignedTo: UserId // processor user
  source: "PRODUCT_COMPLIANCE" | "QSCANAI_FINDING" | "LICENSE_CHECK" | "MANUAL"
  items: CaseItem[] // v1: length always 1
  createdAt: ISO-8601
  sentAt: ISO-8601 | null
  resolvedAt: ISO-8601 | null
}
CaseItem
{
  id: string
  itemType: ItemType // v1: always "COA_EXPIRED" for demo path
  resolutionType: ResolutionType // v1: hardcoded per itemType — UPLOAD_REPLACEMENT_COA
  plProductId: string | null
  vaultCoaId: string | null // the SPECIFIC COA flagged
  finding: string // human-readable description
  severity: "HIGH" | "MEDIUM" | "LOW" // rendered as "High Risk" / "Medium" / "Low"
  status: ItemStatus // see G.2
  latestResponseId: string | null
}
Response
{
  id: string
  caseItemId: string
  resolutionType: ResolutionType // matches CaseItem.resolutionType
  value: ResolutionValue // shape varies by resolutionType (v1: { fileKey, fileName, size })
  submittedBy: { userId, name, email }
  submittedAt: ISO-8601
  status: "SUBMITTED" | "ACCEPTED" | "REJECTED"
  reviewedBy: UserId | null
  reviewedAt: ISO-8601 | null
  rejectionReason: string | null
}

G.2 · State machines

Case status
DRAFT → on Send → OPEN → on item submit → IN_REVIEW → all items accepted → RESOLVED → manual close (v1) / auto-close 90d (v2) → CLOSED
Item status
PENDING → merchant submits → SUBMITTED → processor accepts → RESOLVED
SUBMITTED → processor rejects → PENDING (loops back; merchant re-submits)

G.3 · v1 hardcoding rules (what QP-3593 unlocks later)

v1 itemType → resolutionType map
COA_EXPIRED → UPLOAD_REPLACEMENT_COA // the demo path
PRODUCT_MISSING_COA → UPLOAD_REPLACEMENT_COA // same handler — file upload
COMPLIANCE_FINDING → PROVIDE_URL // stretch: a text input
LICENSE_ISSUE → UPLOAD_LICENSE // stretch: file upload, different copy
