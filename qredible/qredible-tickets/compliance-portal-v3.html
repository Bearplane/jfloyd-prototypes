Compliance Portal v3.1 — End-to-End Vertical Slice

Joseph's v1 implementation slice: build the case-management loop end-to-end across Q-Monitor (processor side) and Merchant Portal (merchant side). Cases hold multiple items of multiple data types (license, website finding, product/COA, KYC); resolution form per item is hardcoded by dataType for v1. This mockup is an engineer-readable spec — Joseph implements in RTQ-Admin-UI / RTQ-Admin-API / Middleware. Every section calls out component boundaries, API endpoints, data shapes, and state transitions.

The Vertical Slice — what gets built end-to-end

Findings from any of three sources can become a case. Cases land in Internal Case Management for review and send-out. Merchants log in, respond, and resolution flows back to the processor. Closing the loop is the v1 success criterion.

1

Finding → Case

"Create Case" action exposed on Q-ScanAI Findings, Product Compliance, and License Check. Selected findings persist as a draft case.

// Route: /qmonitor/cases (list) and /qmonitor/cases/:id (detail panel)
GET /api/qmonitor/cases?status=ALL&merchantId=&assignedTo=
GET /api/qmonitor/cases/:id           → full case incl. items[] (polymorphic), responses[], activity[]
PATCH /api/qmonitor/cases/:id          → status transitions, assignment, SLA edit
POST /api/qmonitor/cases/:id/items     → add item to existing case (multi-source draft building)
POST /api/qmonitor/cases/:id/send      → DRAFT → OPEN
POST /api/qmonitor/cases/:id/responses/:itemId/accept  → accept response, item RESOLVED
POST /api/qmonitor/cases/:id/responses/:itemId/reject  → reject, item back to PENDING

// QP-3591 · Force Close (v1 IN):
POST /api/qmonitor/cases/:id/force-close
  Permission: requires `cases.force_close` (processor admin role)
  Body: { reason: string }   // required, audit-logged
  Side effects:
    1. case.status: ANY → CLOSED
    2. all PENDING items.status → FORCE_CLOSED (terminal)
    3. emit ActivityEvent { type: "CASE_FORCE_CLOSED", reason, actorId, ts }
    4. notify merchant (in-portal banner + email)
    5. write to AuditLog (immutable, separate table)

GET /api/qmonitor/cases/auto-archive-eligible  // cron job target
  // Returns cases where status=RESOLVED and resolvedAt < now - 90d
  // Cron flips them to ARCHIVED; UI hides ARCHIVED unless filter applied

// Tabs render: <ItemsTab/> <ResponsesTab/> <ActivityTab/>
// Items list renders polymorphically — see <ItemRow dataType=...> component map (G.3)

// Processor "Accept Response" (per-item):
POST /api/qmonitor/cases/:id/responses/:itemId/accept
  Side effects (atomic):
    1. response.status: SUBMITTED → ACCEPTED
    2. item.status: SUBMITTED → RESOLVED
    3. if ALL items.RESOLVED → case.status: IN_REVIEW → RESOLVED, set resolvedAt
    4. emit ActivityEvent { type: "RESPONSE_ACCEPTED", caseId, itemId, ts }
    5. emit ActivityEvent { type: "CASE_RESOLVED", caseId, ts }   // if case flipped
    6. notify merchant (in-portal banner + email)

// Processor "Request More Information" (per-item):
POST /api/qmonitor/cases/:id/responses/:itemId/reject
  Body: { reason: string }
  Side effects:
    1. response.status: SUBMITTED → REJECTED
    2. item.status: SUBMITTED → PENDING (back to merchant)
    3. emit ActivityEvent { type: "RESPONSE_REJECTED", reason }
    4. notify merchant in-portal

// Processor "Force Close" (whole case, QP-3591):
POST /api/qmonitor/cases/:id/force-close
  Permission required: cases.force_close (processor admin role)
  Body: { reason: string }   // required
  Side effects (atomic):
    1. case.status: ANY → CLOSED, set closedAt + closedReason
    2. all PENDING items.status → FORCE_CLOSED (terminal)
    3. emit ActivityEvent { type: "CASE_FORCE_CLOSED", reason, actorId, ts }
    4. write AuditLog row (immutable, separate table)
    5. notify merchant by email + in-portal banner

// 90-day auto-archive cron (QP-3591):
// Runs daily at 03:00 UTC
//   UPDATE pl_compliance_case
//      SET status = 'ARCHIVED', archivedAt = NOW()
//      WHERE status IN ('RESOLVED', 'CLOSED')
//        AND COALESCE(resolvedAt, closedAt) < NOW() - INTERVAL '90 days'
// Default Internal CM list filters out ARCHIVED; toggle to include them.

G.2 · State machines

Case status (now includes Force Close + auto-archive)
DRAFT → Send → OPEN → first item submit → IN_REVIEW → ALL items accepted → RESOLVED → cron 90d → ARCHIVED
ANY → Force Close (admin · QP-3591) → CLOSED → cron 90d after closedAt → ARCHIVED
Item status (per-item, independent)
PENDING → merchant submits → SUBMITTED → processor accepts → RESOLVED
SUBMITTED → processor rejects → PENDING (loops back; merchant re-submits)
PENDING / SUBMITTED → case Force Close cascades → FORCE_CLOSED (terminal)
